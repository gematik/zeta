5.3 ZETA Abläufe
Dieses Kapitel spezifiziert die dynamischen Kommunikations- und Prozessabläufe innerhalb der ZETA-Architektur. Um
die Interaktionen zwischen dem ZETA Client, dem ZETA Attestation Service (ZAS), dem ZETA Guard (Authorization
Server) und dem Resource Server nachvollziehbar darzustellen, sind die Unterkapitel chronologisch anhand des
Lebenszyklus einer Client-Instanz strukturiert.
Die Beschreibung folgt dem Weg von stationären und mobilen Clients von der initialen Bereitstellung (Installation und
Start) über die netzwerktechnische Ermittlung der benötigten Endpunkte (Service Discovery) bis hin zur initialen
Schlüsselbindung und Registrierung (Dynamic Client Registration). Darauf aufbauend wird detailliert dargestellt, wie der
Client unter fortlaufendem Nachweis seiner Systemintegrität (Attestierung) Access Tokens bezieht und diese für den

5.3.1.1 Client Installation und Schlüsselgenerierung
Nach der Installation werden beim ersten Start des Primärsystems die Schlüssel generiert, die vom ZETA Client als
Identitäts-Nachweis gegenüber ZETA Guard Instanzen verwendet werden.
Das Ergebnis des Prozesses ist eine initiale Baseline des Clientsystems, die – abhängig vom gewählten
Attestierungsmodus – aus hardwareoder softwaregebundenen Schlüsseln sowie initial erhobenen Zustandsund
Messinformationen besteht. Diese Baseline bildet die Grundlage für alle weiteren Schritte des Attestierungslebenszyklus.
5.3.1.1.1 Schlüsselgenerierung auf Windows und Linux Systemen
Die Installation des Primärsystems (inkl. ZETA Client) erfolgt in der Regel ohne Admin Rechte. Um Zugriff auf die TPM
Funktionen zu erhalten, ist jedoch eine Komponente erforderlich, die unter Admin-Rechten ausgeführt wird. Dafür wird
der ZETA Attestation Service (ZAS) verwendet. Zwischen ZAS und ZETA Client muss eine Vertrauensbeziehung
bestehen, damit nicht beliebige Clients vom ZAS Attestation-Daten anfragen können und damit der ZETA Client nicht mit
einem kompromittierten ZAS kommuniziert.
Das folgende Sequenzdiagramm beschreibt den Prozess der Schlüsselgenerierung und Attestierung auf Windows und
Linux Betriebssystemen. Der Ablauf gliedert sich in einen primären, hardwaregestützten Pfad über das TPM und einen
Software-Attestation Fallback-Pfad.

Das folgende Sequenzdiagramm beschreibt den Registrierungsprozess eines ZETA Clients bei einem ZETA Guard
Authorization Server. Der Ablauf unterscheidet sich grundlegend danach, ob bereits ein ZETA Guard Attestation Token

Wenn der Client noch kein Token besitzt, muss er seine Vertrauenswürdigkeit durch eine Attestierung nachweisen.
Hierfür gibt es drei Unterfälle (innerer alt-Block), abhängig von der genutzten Hardware/Software.
(01)–(10) Hardware-Attestation: Windows / Linux (TPM)
Der ZETA Client sendet einen POST /register Request an den PDP Authorization Server mit folgenden Parametern:
attestation_type = "tpm"
PuK.EK.Enc, C.EK.Enc
PuK.AK.Sig
PuK.Client.Sig
signed_Hash_PuK.Client.Sig
Der Authorization Server validiert die Zertifikatskette von C.EK.Enc sowie signed_Hash_PuK.Client.Sig. Anschließ
end erzeugt er ein CredentialBlob gemäß dem TPM 2.0 Befehl MakeCredential (vgl. TCG TPM Library Part 3,
Section 12.5) und antwortet mit 202 Accepted {CredentialBlob}.
Der ZETA Client leitet das CredentialBlob an den ZAS weiter. Der ZAS führt TPM2_ActivateCredential aus und
erhält das entschlüsselte Secret. Das Secret wird an den Authorization Server via POST /register/verify
{Secret} übermittelt. Nach erfolgreicher Prüfung gelten PuK.Client.Sig und PuK.AK.Sig als hardware-gebunden.

5.3.1.6 Authentifizierung
Die Authentifizierung für Primärsysteme erfolgt per OAuth Token Exchange mit einem SM(C)-B signiertem Subject
Token. Eine erfolgreiche Authentifizierung führt auch zu einem Abschluss der Client-Registrierung, da die Clients
grundsätzlich einem Nutzer zugeordnet werden.
ZETA Guard prüft nicht nur, ob der Nutzer für den Zugriff berechtigt ist, sondern auch ob der Client Zugriff erhalten darf.
Daher muss der ZETA Client neben dem Subject Token auch ein Client Assertion JWT erstellen und mit dem Schlüssel
PrK.Client.Sig signieren. Das Client Assertion JWT dient zur Authentifizierung des Clients gegenüber dem ZETA Guard
Authorization Server und enthält im client_statement die Attestation Daten des Clients (siehe [client-assertion-
jwt.yaml].

5.3.1.7 Zugriff auf den Resource Server
Voraussetzung für den Zugriff auf eine geschützte Ressource (Resource Server) durch einen Client in der ZETA-
Architektur ist, dass der vorgeschaltete Authentifizierungs- und Autorisierungsprozess bereits erfolgreich war und der
ZETA Client über ein gültiges Access Token verfügt.
Durch die Auswertung des OAuth Protected Resource Well-known JSON Dokuments kann der ZETA Client erkennen, ob
eine ZETA/ASL Verbindung aufgebaut werden muss und welche Token benötigt werden.
5.3.1.7.1 Zugriff auf den Resource Server mit ZETA/ASL
Das folgende Diagramm zeigt, wie eine Anfrage vom Fach-Client über den ZETA Client zum Resource Server geleitet
wird.
Das Hauptmerkmal dieses Diagramms ist die Nutzung des ZETA/ASL (Application Security Layer), welcher eine
zusätzliche Verschlüsselungsschicht auf Applikationsebene darstellt. Das Diagramm zeigt in einem großen alt-Block zwei
unterschiedliche Architektur-Ansätze, je nachdem, wo dieser verschlüsselte Tunnel endet (terminiert). Je nach Einstellung
im OAuth Protected Resource Well-known JSON Dokument wird der Tunnel entweder am ZETA Guard HTTP Proxy oder
am Resource Server terminiert.
Vorbedingung: Der ZETA Client hat bereits erfolgreich den Authentifizierungsprozess durchlaufen und besitzt ein gültiges
Access Token für den Resource Server und ggf. für den ASL Tunnel.

5.3.2 Abläufe für mobile Clients
In diesem Unterkapitel werden die Kommunikations- und Attestierungsprozesse für mobile ZETA Clients (z. B. auf Basis
von Android oder iOS) detailliert. Aufgrund der Architektur mobiler Betriebssysteme werden für den Integritätsnachweis
plattformspezifische Mechanismen wie die Android Key and ID Attestation oder die Secure Enclave von Apple genutzt.
Als Fallback-Lösung ist eine Software-Attestierung vorgesehen, falls hardwarebasierte Nachweise nicht erbracht werden
können. Im Unterschied zu stationären Systemen erfordert das mobile Umfeld in der Regel eine interaktive
Nutzerauthentisierung. Daher erfolgt die Autorisierung hier standardisiert über OpenID Connect (OIDC) in Verbindung mit
dem OAuth Authorization Code Flow.
Im Unterschied zu stationären Clients basieren mobile Clients auf plattformintegrierten Sicherheitsmechanismen und
erfordern in der Regel eine interaktive Nutzerauthentisierung. Während bei stationären Clients TPM/ZAS im Fokus steht,
erfolgt der Integritätsnachweis bei mobilen Clients über plattformabhängige Attestierungsverfahren. Die Autorisierung
erfolgt über den OIDC Authorization Code Flow in Kombination mit OAuth 2.0.
Der Lebenszyklus eines mobilen Clients gliedert sich analog zu stationären Clients in folgende Phasen:
Installation und Initialisierung1.
Schlüsselgenerierung2.
Service Discovery3.
Key Preparation und Attestation4.
Dynamic Client Registration (DCR)5.
Authentifizierung (OIDC Flow)6.
Token Exchange und Zugriff7.
Session-Erneuerung8.
Hardware-Attestation SOLL bevorzugt eingesetzt werden und Software-Attestation dient als Fallback-Mechanismus.
 
5.3.2.1 Initialisierung und Schlüsselgenerierung
Nach der Installation der mobilen Anwendung wird beim ersten Start ein initialer Satz kryptografischer Schlüssel erzeugt,
der als langlebige Identität der Client-Instanz dient.
Im Gegensatz zu stationären Clients erfolgt die Schlüsselgenerierung vollständig innerhalb der durch das mobile
Betriebssystem bereitgestellten Sicherheitsmechanismen.

Phase 1: Client Instance Key Pair (PrK.Client.Sig; PuK.Client.Sig)
In dieser Phase wird der Schlüssel, mit dem der Client später Authentifizierungen (Client Assertions) durchführt.
(01): Der ZETA Client bittet das Android OS über den KeyStore, einen neuen primären Schlüssel (Alias_Client_Sig) zu
generieren.
(02)-(03): Das OS reicht den Befehl an die sichere Hardware (TEE / StrongBox) weiter. Dort wird das Schlüsselpaar
erzeugt. Eine entscheidende Neuerung ist hier dokumentiert: (Schlüssel verbleibt in HW). Der private Teil wird niemals in
den Speicher des Betriebssystems oder der App kopiert.
(04): Der Client erhält lediglich eine Referenz (den Alias) auf diesen Schlüssel zurück.
(05): Der Client berechnet lokal den SHA-256 Hashwert über den öffentlichen Teil dieses neuen Schlüssels
(hash_puk_client_sig). Dieser Hash dient in den folgenden Phasen als Bindeglied.
Phase 2: Hardware-Attestierung (Attestation Key)
Hier wird der Hardware-Beweis (die Urkunde) erstellt, dass Phase 1 wirklich in einer StrongBox stattgefunden hat.
(06): Der Client fordert die Generierung eines zweiten Schlüssels an (den Attestation Key, Alias_AK). Dabei übergibt er
den soeben berechneten Hash als AttestationChallenge.
(07)-(08): Die TEE / StrongBox erzeugt das AK-Schlüsselpaar und eine X.509-Zertifikatskette. Die Challenge (der Hash)
wird dabei kryptografisch und unveränderlich in das Zertifikat des öffentlichen AK (PuK.AK.Sig) eingewoben.
(09): Die fertige Zertifikatskette (android_key_attestation_certificate_chain) wird an den Client ausgeliefert.
Phase 3: Expliziter Besitznachweis (Proof of Possession)
Hier beweist das Gerät aktiv, dass es den privaten Attestation Key besitzt.
(10)-(11): Der Client ruft die Signier-Funktion auf und lässt den Hash des Client-Schlüssels (hash_puk_client_sig) mit
dem privaten Attestation Key (PrK.AK.Sig) in der Hardware signieren.
(12)-(13): Die fertige ECDSA-Signatur (signed_hash_puk_client_sig) verlässt die Hardware und wird dem Client
übergeben.
(Zusatzinfo: Durch Phase 2 und 3 kann der Server später verifizieren, dass der Arbeits-Schlüssel unabdingbar mit genau
dieser Hardware verknüpft ist, analog zum Apple-Ablauf).

5.3.2.3 Plattformabhängige Attestierung
5.3.2.3.1 Android
Auf Android-Geräten erfolgt die Attestierung über plattformspezifische Sicherheitsmechanismen, die eine Bindung
zwischen Anwendung, Schlüsselmaterial und Gerät herstellen.
Der Ablauf gliedert sich wie folgt:
(01) Anforderung einer Nonce
Der ZETA Client fordert beim Authorization Server eine Nonce an, um die Aktualität der Attestation sicherzustellen.
(02) Ermittlung von Gerätezustandsdaten
Der ZETA Client sammelt relevante Posture-Informationen:
OS-Version und Security Patch Level
Hersteller und Gerätemodell
Boot-Status und Verified Boot
App-Signaturinformationen
(03) Erstellung des Attestierungsnachweises
Der Client nutzt eine der folgenden APIs:
Android Key Attestation
Android ID Attestation
optional: Play Integrity API
Dabei wird ein kryptografisch signierter Nachweis erstellt, der belegt:
das Schlüsselpaar ist hardwaregebunden
das Gerät befindet sich in einem vertrauenswürdigen Zustand
die Anwendung ist nicht manipuliert
(04) Übermittlung der Attestierungsdaten
Die generierten Attestierungsartefakte werden in ein Client Statement eingebettet und an den ZETA Guard übermittelt.
(05) Validierung im ZETA Guard
Der Authorization Server validiert:
Zertifikatskette des Attestierungsschlüssels
Integrität der Daten
Bindung zwischen Schlüssel und Gerät
Dieser Nachweis stellt einen hardwaregestützten Integritätsbeweis dar und ermöglicht eine hohe Vertrauensstufe für den
Client.

5.3.2.3.2 Apple
Für Apple-basierte Geräte erfolgt die Attestation durch die Secure Enclave in Kombination mit Apple App Attest oder
DeviceCheck.
Der Ablauf ist wie folgt:
(01) Nonce-Anforderung
Der ZETA Client fordert eine Nonce vom Authorization Server an.
(02) Erhebung von Posture-Daten
Der Client liest relevante Systemzustände aus:
Betriebssystemversion
Geräteklasse und Modell
Sicherheitsmechanismen (z. B. Secure Boot, SIP)
(03) Hash-Bildung
Der ZETA Client kombiniert:
Nonce
Posture-Daten
zu einem Hashwert (clientDataHash).
(04) Signatur durch Secure Enclave
Der Client ruft die Apple API (z. B. DCAppAttest.generateAssertion) auf.
Der Hash wird an die Secure Enclave übergeben
Die Secure Enclave signiert den Hash mit dem privaten Schlüssel
Zusätzlich wird ein Counter erzeugt
(05) Erstellung des Attestation Objects
Das Betriebssystem erstellt ein standardisiertes Objekt (Assertion), welches enthält:
Signatur
Zertifikatskette
Authenticator-Daten
(06) Übermittlung an den ZETA Guard
Der ZETA Client überträgt das Attestation Object zusammen mit den Klartextdaten.
(07) Validierung
Der Authorization Server:
validiert die Apple Zertifikatskette
rekonstruiert den Hash
prüft die Signatur und den Counter
Auch hier wird sichergestellt, dass App, Gerät und Schlüssel unverändert und authentisch sind.

5.3.2.3.3 Software Attestierung (Fallback)
Falls keine hardwaregestützte Attestierung möglich ist, wird eine softwarebasierte Attestierung durchgeführt.
Dies betrifft insbesondere:
ältere Geräte
Emulatoren oder spezielle Umgebungen
Geräte ohne verfügbare Attestation-APIs
Der Ablauf ist wie folgt:
(01) Generierung des Client-Schlüssels
Das Schlüsselpaar wird im Software-Kontext erzeugt.
(02) Erhebung von Basisinformationen
Der ZETA Client sammelt einfache Kontextdaten:
Betriebssystem und Version
Architektur
Produkt- und Versionsinformationen
(03) Erstellung einer Challenge-basierten Signatur
Der Client signiert eine vom Server bereitgestellte Nonce mit PrK.Client.Sig.
(04) Erstellung des Client Statements
Die erhobenen Daten und die Signatur werden in ein Client Statement integriert.
(05) Übermittlung an den Authorization Server
Der Client sendet den Nachweis im Rahmen der Registrierung.
(06) Bewertung im ZETA Guard
Der Server kann lediglich prüfen:
Besitz des privaten Schlüssels
Konsistenz der Daten
Eine hardwarebasierte Vertrauensbindung besteht nicht.
Der Resultierende Sicherheitsgrad ist daher geringer.
Die finale Zugriffserlaubnis erfolgt nur unter zusätzlicher Policy-Prüfung.

5.3.3 Dienst zu Dienst Kommunikation
Dieses Unterkapitel widmet sich der maschinellen, nicht-interaktiven Kommunikation zwischen Backend-Diensten
innerhalb der ZETA-Architektur. Um Backend-Workloads sicher zu authentisieren und zu autorisieren, wird das Konzept
der Workload Identity Federation eingesetzt. In diesem Szenario übernimmt der ZETA Guard (Authorization Server) eine
Doppelrolle: Er agiert als Identity Provider (IDP) für den anfragenden Dienst und fungiert als Security Token Service
(STS) bei dem aufgerufenen (zweiten) Dienst. Die Grundlage für diese Vertrauensstellung bildet die Policy Engine,
welche den zentralen Pool der registrierten und vertrauenswürdigen Identity Provider verwaltet und bei der Token-
Ausstellung die Einhaltung der geltenden Zugriffsrichtlinien sicherstellt.

5.3.4 Überblick über den Attestierungslebenszyklus
<< 5.3.4 bis 5.3.12 muss entfernt werden >> Der Attestierungsprozess eines ZETA Clients ist kein singulärer Vorgang,
sondern ein mehrphasiger Lebenszyklus, der sich über die gesamte Nutzungsdauer eines Clientsystems erstreckt. Ziel
dieses Lebenszyklus ist es, zu jedem relevanten Zeitpunkt eine bewertbare und nachvollziehbare Vertrauensgrundlage
für den ZETA Client bereitzustellen.
Der Lebenszyklus beginnt mit der erstmaligen Installation und Registrierung des ZETA Clients und setzt sich über
wiederholte ClientStarts, die Vorbereitung und Bindung kryptografischer Schlüssel, die Erhebung von
Attestierungsnachweisen sowie den Tokenaustausch mit ZETAKomponenten fort. Dabei gilt:
Die initiale BaselineEtablierung schafft einen ersten, definierten Referenzzustand.
Nachfolgende Schritte bauen inkrementell auf bereits etablierten Vertrauensankern auf.
Attestierungsinformationen können sowohl direkt (z.B. bei einem Tokenaustausch) als auch indirekt (z.B. über
gebundene Schlüssel oder DCRArtefakte) in Entscheidungen einfließen.
Der Prozess ist so ausgelegt, dass er sowohl:
hardwarebasierte Sicherheitsfunktionen (z.B. TPM, Secure Enclave) bevorzugt nutzt,
als auch softwarebasierte Verfahren unterstützt, sofern dies erforderlich ist.
Die Tabelle 8 Tab_gemSpec_ZETA_Client_Attestierungsprozess gibt einen Überblick über die Phasen des Attestierungs
und Registrierungsprozesses und ordnet diese den jeweiligen Unterkapiteln und Ablaufdiagrammen zu.
